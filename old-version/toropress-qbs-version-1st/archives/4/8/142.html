nginx反向代理配置+@跳转到www+我的SEO手札

September 24 , 2013 at 18:20 am - - 54点击 - 0 评论 
    server
    {
        listen          80;
        server_name www.veryhour.com veryhour.com;
        if ($host != 'www.veryhour.com' ) {
            rewrite ^/(.*)$ http://www.veryhour.com/$1 permanent;
        }
        location / {
            proxy_pass              http://127.0.0.1:8888/;
            proxy_redirect          off;
            proxy_set_header        X-Real-IP       $remote_addr;
            proxy_set_header        X-Forwarded-For $proxy_add_x_forwarded_for;
            }
    }

这是最简单的反代,关键是@重定向到www都写在一个server段里了,更重要的是目测这种写法对seo比较好~

server {
                server_name test.com;
                rewrite ^(.*) http://www.test.com/$1 permanent;
        }

 

下面这种独立写跳转的情况如果遇上百度,可能就坑死了,为神马呢?

百度的爬虫一直不太智能,譬如遇上你主机有多个域名的时候你就会被坑.

譬如a.com 和b.com同时指向你同一台主机,但是a.com是你的主站,b.com是另一个完全不相干的魂淡指过来你这边的站点.那么这个时候如果百度爬虫过来爬,然后发现了b.com的域名,okay这个时候它就会把这个b.com的这个域名记住,然后接着就开始抽取内容,但是它一定没分析这个时候301跳转了!于是你的a.com就悲剧了,为何?

因为百度记住的是b.com的域名,虽然最后跳转到a.com了,但是它没随跳转而更新域名的习惯和动作,所以你就发现你的内容被百度收录的却是b.com,而压根不是a.com.

 

还有个情况就是,有很多各种零散错误域名的跳转流量到你这里来,很多人都希望收集起来,然后跳转到主域名上去,但因为百度这个爬虫的低智商..最后收录的肯定不是你的主站,所以最好有个中间跳转的动作,譬如从c.com d.com f.com等外部站点过来的流量,我们先集中到一个第三方站点k.com上面,然后我们不要用nginx跳转,我们在k.com上面输出js命令客户端自行跳到a.com,这样流量最终就会汇聚到a.com了,但是收录什么的这时候应该是没有影响,为何?

百度一般是不解析js的,那么最终到了k站之后,百度的爬虫就会停下来,收录的是k.com的链接,但是k.com是我们故意拿出来挡在前面做牺牲的,这就无所谓了.但最终真实用户的流量也回到我们a.com上面才是实实在在的利益~  

至于这些外链的价值就只能要他的流量,引用就不要想了,如果要引用,除非是你的站点权限足够引起百度的注意吧,不然这种张冠李戴的情况我暂时是推荐这样的做法.
